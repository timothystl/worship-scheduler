TLC ChMS — Operator Reference Manual

Timothy Lutheran Church • Last updated: April 25, 2026

1. Introduction & Access

The TLC ChMS (Church Management System) is a web application built for Timothy Lutheran Church staff and volunteers. It manages people records, households, giving, attendance, the church register, and communications — all in one place, synced with Breeze ChMS where applicable.

Access

URLhttps://chms.timothystl.org
Old URLvolunteer.timothystl.org/chms — redirects automatically
PlatformCloudflare Workers + D1 (SQLite). No server to maintain; runs globally on Cloudflare's edge network.
BrowserAny modern browser (Chrome, Firefox, Safari, Edge). No app to install.

What this manual covers

This reference covers every tab and feature available to operators. It assumes you are logged in with at least staff role access. Some features (Giving entry, Brevo sync, user management) require finance or admin role — those are noted throughout.

Note: The app is a single page — all tabs load without a full page refresh. If something looks wrong after returning from another app, try pressing F5 to reload.

4. People

The People tab is the core of ChMS. It lists all active people, with search, filters, and access to individual profiles.

Browsing the list

By default, the list shows Members only (people with member_type = member). Click the Members toggle button in the toolbar to switch to all active people. Archived and deceased people are hidden unless you click Archived.

Use the search box to filter by name (first or last). The list updates as you type.

Filtering by missing data

Click Filter to open the missing-data filter panel. Checkboxes are organized by category (contact info, dates, sacraments, etc.). Selecting multiple boxes applies AND logic — results must be missing all selected fields. This is useful for data cleanup drives (e.g., "show me all members missing both email and phone").

Adding a person

  1. Click + Add Person.
  2. Fill in at minimum: first name, last name, member type.
  3. Optionally assign to a household — type in the household field to search existing households, or leave blank to create a standalone record.
  4. Click Save.
Note: Adding a person here does not automatically push them to Breeze. See the Push to Breeze action below.

Person profile

Click any person's name to open their profile. The profile is divided into collapsible sections that can be edited inline.

Photo upload

  1. Open a person's profile and click the photo area (or the camera icon).
  2. Select an image file. A crop tool appears.
  3. Drag and resize the crop box, then click Save Photo.

Photos are stored in Cloudflare R2 and served via /admin/r2photo/. The crop step ensures consistent square thumbnails.

Archive, deceased, and reactivate

At the bottom of a person's profile:

  • Archive — sets status = archived. The person disappears from the default list but remains in the database and in giving history. Use for people who have moved away or become inactive.
  • Mark Deceased — sets status = deceased and records death_date. Deceased people are excluded from birthday/anniversary cards and email/SMS sends.
  • Reactivate — restores an archived or deceased person to status = active. Appears when viewing an archived profile (use the Archived toggle on the people list to find them).
Warning: There is no undo for archive/deceased actions in the UI. If you mark the wrong person, open their profile via the Archived toggle and click Reactivate.

Mark Seen Today

Click Mark Seen Today on any profile to set last_seen_date to today's date. This feeds the engagement queue and the Contact Completeness report.

Follow-ups

Click + Follow Up on a profile to add a follow-up item with a due date and notes. Open follow-ups appear on the Dashboard card. Mark them complete from the profile or the dashboard.

Breeze integration per-person

Two sync actions are available on each profile:

ButtonWhen to useWhat it does
Push to Breeze Person has no breeze_id (new person entered here) Creates the person in Breeze via API; saves the returned Breeze ID locally.
Sync Breeze Person already exists in Breeze (has a breeze_id) Pulls current data from Breeze and overwrites local fields. Use after editing a record in Breeze directly.
Note: Sync is currently one-way pull from Breeze → ChMS for existing records, and one-way push ChMS → Breeze for new records. Full bidirectional sync is not yet implemented.

SMS opt-in

The sms_opt_in checkbox on a profile controls whether this person receives automated birthday/anniversary SMS messages. Only check this if the person has explicitly consented to text messages from the church. Phone number must be a valid US number.

Directory privacy settings

The public_directory flag and the three dir_hide_* checkboxes let individuals control what appears if a member-facing directory is published. These flags are currently stored but not yet surfaced in a public-facing directory view.

5. Households & Directory

The Households tab provides a directory-style view of families and organizations. Each household groups its members together and can have a shared photo and address.

Households list

Households are shown as cards with the household name, photo (if uploaded), and member names. Click any card to open the household detail modal.

Household detail modal

Shows all members of the household with their family_role. From here you can:

  • Edit the household name, address, and notes
  • Upload a household photo (separate from individual profile photos)
  • Add or remove people from the household
  • Click any member name to jump to their profile

Household naming

Household names are auto-generated from the head's last name (e.g., "Smith Family"). When two households share the same last name, the system disambiguates automatically using the head's first name (e.g., "John Smith Family" vs "Mary Smith Family"). You can override the name by editing it directly in the household modal.

Organizations

Organizations are a special household type for non-family groups (e.g., businesses, ministries). They appear in the same list but are visually distinct. Organization records can hold contact info and be associated with people in the same way as households.

Tags

Tags can be applied to people and are managed in Settings → Tags. Common uses include ministry involvement (choir, usher, Sunday school), small groups, or giving envelope assignment. Tags sync with Breeze tag fields when Breeze sync is run.

Household head requirement

Every household should have exactly one person with family_role = head. The system uses the head's name for household display and for generating anniversary cards. If a household is headless, use Settings → Fix Household Heads to auto-promote the spouse or oldest member.

Note: The Settings card shows a live count of headless households. Check it after bulk imports or when adding new families.

Household photo

Click the photo area in the household detail modal to upload an image. Household photos appear on the Households tab cards. If no photo is uploaded, a placeholder with the family initial displays instead.

6. Giving

The Giving tab is available to finance and admin roles only. It manages contribution records, batches, and synchronization with Breeze.

Giving model

Contributions are organized into batches — groups of gifts received together (typically a single Sunday's offering). Each gift record belongs to exactly one batch and one person.

Amounts are stored in the database as integer cents (e.g., $25.00 is stored as 2500). The UI always displays dollars. Never enter amounts in cents manually.

Batches

Batches have three states:

StateMeaning
OpenEntries can be added, edited, or deleted. Normal state during data entry.
ClosedLocked. No edits allowed until reopened. Use once a batch is verified and balanced.
(Reopened)A closed batch that was reopened for correction. Returns to Open state.

To create a new batch: click + New Batch, enter a name (typically the date, e.g., "April 27, 2026") and the batch date, then click Create.

To close a batch: open it and click Close Batch. To reopen: click Reopen on a closed batch.

Adding gifts to a batch

  1. Open the batch (click its row in the batch list).
  2. Click + Add Gift.
  3. Fill in: Person (search by name), Fund, Amount (in dollars), Method, Date.
  4. Optional: Check #, Notes.
  5. Click Save.

Gift fields

FieldValues / Notes
personMust match an existing person in the database (type to search)
fundSelect from configured funds (see Funds below)
amountDollar amount (e.g., 25.00). Stored as cents internally.
methodcash, check, card, transfer, online, other
check_numberOptional; for check gifts
dateDate of the gift (defaults to batch date)
notesFree text; any special instructions or designations

Editing individual gifts

From within an open batch, click any gift row to open an edit modal. You can change any field except the person. To reassign a gift to a different person, delete and re-enter it.

You can also access a person's giving history from their profile — click a batch number in their giving list to open that batch, then click a gift row to edit it.

Quick entry

Click Quick Entry in the Giving tab toolbar. This mode auto-creates a batch for the current month if one doesn't exist, and lets you enter multiple gifts rapidly without navigating the batch list. Useful for ACH/online donations that arrive mid-month without a specific service.

Funds

Funds represent giving designations (e.g., General Fund, Building Fund, PNG Mission). They are configured in Settings → Funds. Fund names from Breeze are synced during import and follow Breeze's numeric naming convention (e.g., "40085 General Fund"). Sub-fund variants (e.g., "40085 Christmas Offering") are stored separately but grouped by numeric prefix in the Giving by Fund report.

Breeze giving sync (pull from Breeze)

This imports contribution records directly from Breeze's API for a given date range.

  1. Go to Settings → Breeze Giving Sync.
  2. Enter a start and end date.
  3. Click Sync Now.

The sync:

  • Imports all Breeze gifts in the date range that aren't already in ChMS (matched by Breeze payment ID).
  • Runs an orphan-cleanup pass: gifts previously imported from Breeze that no longer appear in Breeze's giving list for the same person+date are removed if a current replacement exists. This handles Breeze's behavior of creating new payment IDs when a gift is edited.
  • Has a 45-day grace window: December contributions entered in Breeze during January are imported with their actual December date.
Warning: Always export a D1 backup before running a large sync. See Troubleshooting → Before risky operations for the backup command.

CSV import from Breeze export

For bulk historical imports, you can import giving records from a Breeze-exported CSV file.

  1. Export giving from Breeze as CSV (Contributions → Export).
  2. In ChMS, go to Settings → Import → Giving CSV.
  3. Upload the CSV file and click Import.

Known CSV quirks (handled automatically)

  • Split-fund donations: One payment spread across multiple funds appears as multiple rows with the same Payment ID. ChMS tracks these by position (pid, pid-2, pid-3) so each fund allocation is preserved.
  • Float person IDs: Breeze sometimes exports person IDs as 43826663.0 — the .0 is stripped automatically.
  • "nan" fund name: Blank fund names from Excel export become the text nan — mapped to General Fund automatically.
  • Negative amounts: Valid for refunds/adjustments; imported as-is.

After import, a summary shows the count of imported gifts and an expandable list of any skipped Payment IDs (with reasons).

Reconciliation tools

Two tools help when ChMS totals don't match Breeze's totals for a period:

ToolWhat it does
Reconcile OrphansFetches Breeze's giving list for the date range and removes ChMS entries whose Breeze payment ID no longer exists in Breeze — only if a current replacement exists for the same person+date. Safe to run repeatedly.
Diagnose (admin only)Read-only analysis: classifies every ChMS entry as "valid" (Breeze ID still in Breeze), "orphan" (Breeze ID missing from Breeze), or "manual" (no Breeze ID). Shows per-fund extra totals and twin-row detection. Use to understand a discrepancy before deciding how to fix it.
Force Remove N (admin only)Deletes orphan entries without requiring a replacement. Requires entering a matching count and cents total as a safety guard. Use only after Diagnose confirms the entries are truly stale.

Exporting giving data

In Settings → Export, you can download:

  • Giving by year — CSV of all gifts for a selected year (for statements, audit, etc.)
  • People — CSV of all person records
  • Register — CSV of church register entries

7. Attendance

The Attendance tab tracks weekly worship service counts. It records service-level totals, not individual check-ins — there is no per-person attendance record in the current system.

Adding an attendance record

  1. Click + Add Service.
  2. Select the date and service time (e.g., 8:00 AM, 10:30 AM).
  3. Enter the attendance count.
  4. Optionally set the service type: regular (default), special (Christmas, Easter, etc.), or midweek (Advent, Lent, etc.).
  5. Click Save.

For Sundays with two services, add each service separately — the dashboard's "Last Worship" card will combine them automatically.

Attendance table

The table shows all recorded services sorted newest-first. Click the collapse/expand toggle to reduce the table to a compact view. Click any row to edit the entry (correct a count, change the type, etc.).

Attendance chart

Below the table, a line chart plots Sunday regular attendance over time (oldest on the left). Key features:

  • Easter and Christmas markers — dashed vertical lines mark Easter Sunday and Christmas Eve/Day each year.
  • Special/Midweek chart — a separate bar chart below shows special (amber bars) and midweek (purple bars) services. These are excluded from the Sunday average and trend line.
  • Drag to resize — drag the bottom edge of either chart to make it taller or shorter.

Year-over-year comparison

In the Reports tab → Attendance by Service tile, you can switch to a multi-year view. Select up to five years via checkboxes; the chart shows grouped bars by service time with one bar per year. This is useful for presenting attendance trends to the council.

Note: Midweek and special service counts do not affect the Sunday average shown on the Dashboard card. The Dashboard card always shows regular Sunday services only.

Adding a register entry

  1. Click + Add Entry.
  2. Select the event type.
  3. Search for and select the person (must already exist in the People tab).
  4. Enter the event date and any type-specific fields (officiant, witnesses, etc.).
  5. Click Save.

Editing and deleting entries

Click any register row to open it for editing. Corrections (wrong date, misspelled name) can be made at any time. Only admin users can delete a register entry — this is intentional, as the register is a legal record.

Relationship to person profile

Adding a baptism, confirmation, or marriage record in the register does not automatically update the corresponding date fields on the person's profile (baptism_date, confirmation_date, anniversary_date). Update those fields on the profile separately if you want them to appear in reports and anniversary cards.

Exporting the register

Go to Settings → Export → Register to download all register entries as a CSV. This is suitable for sharing with the LCMS district or for off-site backup.

9. Reports

The Reports tab provides analytical views of people, giving, and attendance data. Finance reports require finance or admin role; all other reports are available to staff and above.

Membership Summary

Shows a breakdown of the congregation by member type (member, visitor, friend, etc.) with counts and share percentages. Also includes an Age Group table (Under 18, 18–29, 30–44, 45–64, 65+, Unknown) with count and share for each group. Useful for council presentations and LCMS statistical reports.

People Insights

A deeper demographic view with six panels:

  • New contacts by month — bar chart of the last 24 months (based on first_contact_date)
  • New people by year × type — cross-tabulation of new arrivals by member type
  • Age distribution — bars for 6 age buckets
  • Gender — pie chart
  • Household composition — bars: single / couple / small family / large family / unassigned
  • Sacramental pipeline — members only: neither sacrament / baptized only / confirmed only / both

Contact Completeness

Shows what percentage of active people (or members-only, via a scope toggle) have each key field filled in: email, phone, address, date of birth, and photo. Each row is a progress bar — green means complete.

Click any row to drill down to the list of people missing that field. Each name is clickable, taking you directly to their profile to fill in the gap.

Giving by Fund

The primary year-end and council finance report. Enter a date range and click Run. Shows:

  • Total amount per fund, grouped by numeric prefix (e.g., all "40085 *" variants collapse into one group with a subtotal)
  • Overview tiles: Total Givers, Total Gifts, Grand Total, Avg / Gift, Avg / Giver
  • Age group breakdown: givers, gifts, total, and share per age bucket
  • Reconcile Orphans button — see Giving section for details
  • Diagnose button (admin) — detailed entry classification

Giving by Method

Shows giving totals broken down by payment method (cash, check, card, transfer, online, other) for a date range. Includes a pie chart of share by method and a data table. Useful for treasurer reconciliation.

Giving Insights

Select a year to see four analytical panels:

  • Top 25 givers — ranked by total giving, with clickable names linking to their profiles (finance/admin only)
  • Lapsed givers — people who gave in the prior year but have not given in the selected year; sortable by prior-year total
  • Giving frequency distribution — histogram of givers by number of gifts (1 / 2–5 / 6–12 / 13–26 / 27+ per year)
  • 5-year trend — table of givers, gifts, total, avg/gift, avg/giver for the last five years

Giving Trend (Year-over-Year)

Overlapping line chart comparing weekly giving for the current year vs. prior year on the same axes. Dashed vertical lines mark Easter (computed per year using the Meeus/Jones/Butcher algorithm) and Christmas. Drag the bottom edge to resize the chart.

Giving × Attendance

Overlay chart with green bars for weekly attendance (left axis) and a teal line for weekly giving (right axis), aligned by Sunday date. Overview stats include correlation coefficient with a qualitative label (Strong+, Moderate+, Weak+, None, etc.). Useful for identifying whether giving and attendance move together over time.

Attendance by Service

Two views accessible via toggle:

  • Date range — bar chart of attendance by service time for a custom period
  • Multi-year — grouped bar chart; select up to 5 years via checkboxes (2 most recent pre-checked). Each cluster shows all service times; one bar per year per time. Good for council slide decks.

Engagement Queue

Lists people who may need pastoral attention based on inactivity signals (no last_seen_date in many months, open follow-ups, etc.). The queue is read-only; action is taken by visiting the person's profile and using Mark Seen Today or + Follow Up.

Note: All reports use live data from D1. There is no cache — large date ranges (multi-year) may take a moment to load on slower connections.

10. Settings

The Settings tab consolidates administrative functions. Most sub-sections require admin role; exceptions are noted.

User management (admin only)

Add, edit, and remove ChMS user accounts.

  • + Add User — creates a new account. Required: username, password, role.
  • Edit on an existing user — change their role or reset their password. You cannot change a username; delete and re-create if needed.
  • Delete — permanently removes the account. The user's audit trail remains; only the login is removed.
Warning: There is no confirmation email for new accounts. Give the new user their username and initial password directly and ask them to note it immediately.

Tags

Add and remove tag definitions used across People profiles. Tags are free-text labels (e.g., "Choir", "Usher", "Sunday School Teacher"). Removing a tag definition here does not remove it from people who already have it — clean up assignments on individual profiles first.

Member types

The five member types (member, visitor, friend, associate, inactive) are fixed system values and cannot be renamed or added to via Settings. Contact a developer to add a new type.

Breeze people sync (admin/finance)

Pulls all people records from Breeze and upserts them into ChMS. Use this for an initial bulk load or after making many changes in Breeze.

  1. Click Sync People from Breeze.
  2. The sync runs in batches. A progress counter updates as pages complete.
  3. New people are created; existing people (matched by Breeze ID) have their fields updated.
Note: People-sync does not delete anyone from ChMS if they were removed from Breeze. Manage removals manually (archive or delete on the profile).

Breeze tag sync (admin/finance)

Fetches all tag definitions from Breeze and creates matching tag records in ChMS. Run this before a people sync to ensure tags are available for assignment.

Breeze giving sync

See Section 6 — Giving → Breeze giving sync for full details. The date-range entry fields are here in Settings.

Brevo newsletter sync (admin only)

Brevo is the church's email newsletter platform. ChMS can sync member email addresses to a Brevo contact list.

  • Reconciliation view — shows a comparison of active members in ChMS vs. contacts in the configured Brevo list. Highlights people in ChMS but not in Brevo ("missing from Brevo").
  • Add All Missing — pushes all missing members to Brevo in one operation.
  • Individual profiles also have an Add to Newsletter button to add a single person.
  • Auto-sync: when a member's email address is changed and saved on their profile, ChMS automatically updates their Brevo contact record.
Note: Only people with member_type = member and a valid email address are eligible for Brevo sync. Visitors and friends are not synced to the newsletter list automatically.

Birthday & anniversary email test (admin only)

Two buttons let you verify that the automated email/SMS system is working without waiting for an actual birthday:

  • Send Test Birthday Email — sends a sample birthday email to the admin's own email address.
  • Send Test Anniversary Email — sends a sample anniversary email.

If the test email doesn't arrive within a few minutes, check with a developer to verify the RESEND_API_KEY and EMAIL_FROM secrets are correctly set.

Fix household heads (admin only)

Displays the count of households that have no member with family_role = head. Click Fix Household Heads to automatically promote the first available spouse, or the first member if no spouse exists, to head role. Safe to run at any time.

Import (admin/finance)

Sub-section for bulk data imports:

  • Giving CSV — import a Breeze-exported CSV of contribution records (see Section 6)
  • Register CSV — bulk-import register entries from a formatted CSV

Export

Download data as CSV files. Available exports: People, Giving (by year), Register. See Section 6 for details.

11. Things That Update Automatically

Several processes run automatically without any operator action needed. This section explains what they do and when, so you know what to expect.

Daily cron — 9:00 AM Central

A scheduled job runs every morning at 9:00 AM Central time. It performs four tasks in sequence:

TaskWhat happens
Birthday emails & SMS Finds all active, non-deceased people whose dob matches today's month and day. Sends a birthday email via Resend. Sends an SMS via Brevo to anyone with sms_opt_in = true and a valid phone number. Deduplication prevents double-sends if the cron runs more than once in a day.
Anniversary emails & SMS Finds active couples whose anniversary_date matches today. Where both spouses share a household and the same email, one combined email is sent. SMS goes to opted-in phone numbers. Deceased spouses are excluded.
Brevo auto-sync This runs on save, not at 9am — see below.
Audit log pruning Deletes old audit log rows to keep the table manageable: birthday/anniversary send records older than 60 days are removed; all other audit entries older than 365 days are removed.
Note: If a birthday or anniversary falls on a day the cron fails (rare Cloudflare outage), the email/SMS will not be re-sent retroactively. You can trigger a manual test send from Settings, but it goes to your own address, not the member's.

Brevo auto-sync on email save

Whenever a member's email address is changed and saved on their profile, ChMS immediately pushes the updated email to Brevo. This ensures the newsletter list stays current without a manual bulk sync. Only members (not visitors/friends) trigger this auto-sync.

Website intake forms

Two forms on the church website feed data into ChMS automatically:

  • Connect Card (website contact form) — creates a new Visitor person record and a follow-up item in ChMS. Staff see the new person on the Dashboard "New Contacts" card and in the Follow-Ups list.
  • Prayer Request form — creates a prayer request row in ChMS. It appears on the Dashboard Prayer Requests card immediately.

No operator action is needed to receive these — just check the Dashboard cards regularly.

What does NOT update automatically

  • Breeze people sync (must be triggered manually in Settings)
  • Breeze giving sync (must be triggered manually)
  • Brevo bulk newsletter reconciliation (must be triggered manually)
  • D1 database backups (Cloudflare PITR runs automatically for 30 days, but manual exports must be done by a developer before risky operations)

12. Asking Claude to Make Changes

ChMS is built on Claude (Anthropic's AI), and changes to the app are made by describing what you want in plain English. This section explains how to write prompts that get results quickly and correctly.

The golden rule

Be specific about where (which tab, which button, which report) and what (exactly what should change, and what the result should look like). Vague prompts produce vague results.

Good prompt examples

In the Giving tab, when closing a batch add a confirmation dialog that shows the batch's total amount and entry count before the close happens.
On the People list, add a "Last Seen" column showing the last_seen_date value for each person. It's already stored in the database — just needs to be displayed in the list table.
In the Reports tab, the "Giving by Fund" date range defaults to January 1 of the current year. Change the default end date to today instead of December 31.
The birthday email subject line currently says "Happy Birthday from Timothy Lutheran." Change it to "Happy Birthday from your TLC family!"
Add a "Notes" field to the Attendance entry form so we can record the preacher's name or a special event note alongside the count.

Bad prompt examples — and why

Fix the giving thing.

No location, no description of what's wrong, no expected behavior.

Make it better.

Completely unactionable — "better" means different things to everyone.

The reports are slow.

13. Requires a Developer

Some changes to ChMS are outside what operators can do through the UI and require a developer (someone with access to the codebase, Cloudflare dashboard, and wrangler CLI). The following tasks fall into this category.

Email template text

The body and subject of birthday and anniversary emails (and SMS messages) are defined in src/api-emails.js. Changing wording, adding a signature, or adding a new template requires editing that source file and deploying.

Breeze field mappings

The mapping between Breeze's custom field IDs and ChMS's person record columns is hardcoded in src/api-import.js (and the Breeze API client in src/breeze.js). If Breeze changes a field ID or you want to map an additional Breeze custom field to a ChMS column, a developer must update those files.

Database schema changes

Adding a new column, table, or index requires:

  1. Creating a migration file in migrations/NNNN_description.sql
  2. Adding the same statement to the migrations array in src/db.js (for cold-start safety)
  3. Running wrangler d1 migrations apply tlc-volunteer-db --remote

The migrations system tracks which have been applied; re-running is safe.

API key rotation

All API keys and credentials are stored as Cloudflare Worker secrets (not in the codebase). To rotate a key:

  1. Log into the Cloudflare dashboard → Workers & Pages → tlc-chms → Settings → Variables & Secrets.
  2. Update the secret value.
  3. The new value is live immediately — no redeploy needed.
Secret namePurpose
ADMIN_PASSWORDBreak-glass admin fallback (not a regular account)
RESEND_API_KEYResend.com — birthday/anniversary email delivery
EMAIL_FROMSender address for Resend emails
BREVO_API_KEYBrevo — newsletter sync + transactional SMS
BREVO_LIST_IDBrevo contact list ID for newsletter sync
BREEZE_API_KEYBreeze ChMS REST API key
BREEZE_SUBDOMAINBreeze account subdomain (e.g., tlc)

wrangler.toml changes

wrangler.toml controls the Worker name, D1/KV/R2 bindings, cron schedule, and routes. Any changes here (e.g., adding a new KV namespace, changing the cron schedule, adding a custom domain) require a developer and a redeploy.

Deploying the app

Deploy the production Worker with:

wrangler deploy

Deploy to staging with:

wrangler deploy --config wrangler.staging.toml

Staging is at breeze-proxy-worker-staging.timothystl.workers.dev/chms. Test risky changes there first.

Cron schedule

The daily cron is set to 0 14 * * * (UTC) = 9:00 AM Central Standard Time / 10:00 AM CDT. Changing the time requires editing wrangler.toml and redeploying.

Adding a new tab or major feature

The entire frontend is in src/html-chms.js. New tabs must be added to the showTab() labels map and have a corresponding load function. New API routes go in the appropriate domain module under src/ and must be dispatched from src/api-chms.js or src/api-admin.js.

Note: See SECRETS.md in the repository root for a full inventory of all secrets, their rotation steps, and who has access.

14. Troubleshooting

Birthday/anniversary email didn't send

Check first: Is the person marked status = active and not deceased? Is their dob or anniversary_date filled in? Is the date exactly today (month and day)? The cron fires at 9 AM Central — if it's still morning, it may not have run yet.

Test send: Go to Settings and use the Send Test Birthday Email button to confirm Resend is working. If the test doesn't arrive within 5 minutes, a developer needs to check the RESEND_API_KEY secret.

Brevo sync not working

Confirm the person has member_type = member and a valid email. Check Settings → Brevo → Reconciliation view to see if they appear in the "missing" list. If the bulk Add All Missing button completes but the count doesn't change, a developer should check the BREVO_API_KEY and BREVO_LIST_ID secrets.

Giving totals don't match Breeze

This is the most common issue. Work through these steps in order:

  1. Run Giving by Fund → Reconcile Orphans for the affected date range.
  2. If still mismatched, run Giving by Fund → Diagnose (admin) to classify all entries.
  3. Review the "orphan" list — these are ChMS entries whose Breeze payment IDs no longer exist in Breeze.
  4. If all orphans are confirmed stale (deleted from Breeze intentionally), use Force Remove (admin) with the matching count and cents guard.
Warning: Export a D1 backup before running Force Remove. Once deleted, entries can only be recovered from the backup.

Person appears twice in the list

This can happen if a person was added manually in ChMS and then imported from Breeze as a new record. To merge:

  1. Identify which record is the "correct" one (the one with the right Breeze ID, if applicable).
  2. Move any giving entries, register records, and follow-ups from the duplicate to the correct record (currently manual — a developer can do this via D1 queries if there are many).
  3. Archive or delete the duplicate.

Photo not displaying

R2 photos are served via /admin/r2photo/. If a photo uploaded successfully but isn't displaying, try a hard reload (Ctrl+Shift+R or Cmd+Shift+R). If still missing, the photo key may not have been saved to the person record — re-upload it on the profile.

App is slow or unresponsive

Cloudflare Workers have a 30-second execution limit. If a request times out (you see a Cloudflare error page instead of the app's error message), it's likely a large sync or report query. Try:

  • Narrow the date range on reports or giving sync
  • If doing a bulk Breeze people sync, the frontend re-triggers automatically in pages — let it complete rather than refreshing

If the app itself won't load (not just a slow query), check cloudflarestatus.com for an outage.

Before risky operations — take a backup

If a developer is about to run a bulk sync, force-remove, or migration, export a D1 snapshot first:

wrangler d1 export tlc-volunteer-db --remote --output pre-op-backup-$(date +%Y%m%d-%H%M).sql

To restore from that backup:

wrangler d1 execute tlc-volunteer-db --remote --file pre-op-backup-20260425-0900.sql

Cloudflare also keeps point-in-time backups for 30 days — go to Cloudflare Dashboard → D1 → tlc-volunteer-db → Backups to restore a specific timestamp.

Headless household count keeps increasing

Newly imported people from Breeze may not have a family_role set. After any Breeze people sync, check Settings → the headless count indicator and run Fix Household Heads if the count is high.
